# cs repo を PD 論の部品として引用する原則

project-design（pd）の「プロジェクトデザイン論」は、creation-space（cs）で独立に構築される「創造のモデル」を**部品として引用**して構築される。

## 目的階層

```
原典 30 領域調査（cs の部品の粒度）
        ↓
cs repo の目的: 「創造のモデル」を独立して確立すること
        ↓
pd repo の最終ゴール: cs を部品として積み上げ、
                     「プロジェクトデザイン論」を構築すること
```

（他の部品として `kesson-driven-thinking` の「欠損駆動思考」等も同様の位置付け）

## 原則

1. **cs は独立構築**: cs は pd のために存在するが、pd のスケジュールや構造に従属しない。cs 自身の論として自律的に構築される
2. **引用の向き**: PD 論は cs の成果物を**引用**する（pd → cs の一方向）。cs は pd に対して出力義務を持たない
3. **参照の向き**: cs 側は原典調査の品質チェック用として pd wiki を**参照**する場合がある（cs → pd の読み取り）。参照は引用とは別レイヤーで、必須ではない
4. **内容の独立性**: cs 側の原典要約・解説と pd wiki の内容が**部分的に重複することは許容**。それぞれ独立した文脈（cs=モデル確立の調査、pd=体系化された読者向け）で存在する

## PD 論からの引用ルール

PD 論の文書で cs の成果を引用するとき:

- **粒度**: 領域レポート（`cs/transform/domains/publish/domains/report-D{NN}-*.md`）単位、または領域横断的な概念単位で引用する
- **表記**: 引用元を明示する（例: `creation-space D03 化学 領域レポート §5-段階` 等）
- **cs の更新追随**: cs の成果が改訂された場合、PD 論側でも引用箇所を更新するか、引用時点の版を明記する

## cs の構造変更への非介入

cs の内部構造（`knowledge/` 配下のディレクトリ名、ファイル命名、調査フロー等）は cs 側の自律に任せる。pd 側から強制しない。

- 例外: 共通 hooks / rules / 運用パターンの見本（`.claude/rules/reference-repos.md` 参照）は pd が提供するが、cs の**理論内容**には介入しない

## pd 側の位置付け整理（CLAUDE.md 関連リポジトリ）

従来「PD の実践事例」と表現していた `creation-space` を、「PD 論の基盤部品として引用する『創造のモデル』の独立構築拠点」に読み替える。
