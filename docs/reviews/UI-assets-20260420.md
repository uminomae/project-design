## index.html

Project Design 理論のランディング。Hero（Project Design）→ Overview → Doing → Being → Direction の単一ページ構成。3 repo（kesson / creation / awareness）とは **異なるデザイン言語**（light theme + glow 表現）。

### レイアウト
- **単一ページスクロール**: menu-toggle dropdown + scroll-hint のみ、多層 navigation なし。KEEP — ランディングとしての潔さ
- **`body > .top-bar + #webgl-container + #content.page-shell > main.container.page-main`**: page-shell が z-index 1、webgl が背景の 2 層。KEEP
- **hero + 3 major sections（Overview/Doing/Being）+ Direction**: Doing/Being の二項対立を section で配置、最後に 3 repo への導線。KEEP — 理論の骨格とページ構造が一致
- **`.pills` + `.pill--doing / --being`**: Doing カテゴリの概念リンク集（欠損駆動 / 美駆動 / 感情処理 / 瞑想）を pill で並べる。KEEP — 軽量 tag-like link の別表現
- **`.repo-grid`**: kesson / creation / awareness へのカードグリッド、3 repo 俯瞰の入口。KEEP — 単一ソース設計に直結する
- **4 modal（about / knowledge / howto / slides）**: modal 名で機能区別、`data-modal-open / data-modal-close` 属性駆動。KEEP — 軽量 modal pattern、SYSTEMIZE（他 repo 展開候補）

### アニメーション
- **webgl-container の Three.js 背景**: 他 repo と共通だが、pd は `#webgl-container` 命名。KEEP（命名差は残す必要性薄い、SYSTEMIZE 候補）
- **`glow-heading / glow-text / glow-card / glow-link`**: pd の署名的表現。CSS で発光効果を RGB + radius + opacity 可変で実装。KEEP — pd 固有のブランド
- **scroll-hint.visible class toggle**: JS で条件付き表示。KEEP
- **modal fade + backdrop blur（推定）**: modal 開閉のアニメ。KEEP

### 配色
- **`--bg: #fafaf9`（ほぼ白）**: 3 repo（`#050508` ほぼ黒）と**完全に反転**。KEEP — pd の立場性を示すブランド判断
- **`--ink: #0f172a`**（暗いネイビー本文色）: 読みやすさの軸。KEEP
- **`--gold / --navy / --coral`**: 3 色の中性アクセント。3 repo の accent 青（`100, 150, 255`）と系統が違う。KEEP — **pd 固有の temperature palette**（warm 系を含む）
- **`--glass-bg-strong/medium/soft/faint`**: 白ベースの 4 階層 transparency。glassmorphism。KEEP
- **`--glow-r/g/b` 実行時可変**: `?dev` panel で RGB を GUI 変更できる。KEEP — **design system を live で試す仕組み、他 repo にない**
- **`--glow-h-r1/2/3 (heading) / --glow-t-r1-5 (text) / --glow-c-r1/2/3 (card)`**: glow の半径を要素種類ごとに 3-5 段のリングで分ける。KEEP — 微妙な光彩表現の細分化
- **`--card-bg-alpha / --card-blur / --card-border-alpha`**: カード外装も runtime 可変。KEEP

### タイポグラフィ
- **三本柱: `--serif-en` (EB Garamond) / `--serif-jp` (Noto Serif JP) / `--sans` (Noto Sans JP)**: 3 repo と重なるが、**英文用の別 serif 指定**が独自。KEEP — pd の英文比率が高いため意図的
- **glow を見出しに multi-layer text-shadow 付与**: 発光表現。KEEP
- **`data-i18n-key` + `data-i18n-aria-label-key` 方式**: kesson（data-ja/data-en）/ creation・awareness（data-i18n + data-i18n-attr-*）と**さらに別方式**。key + dictionary 参照。KEEP（実装上 aria-label も含めて i18n 化できる点で高機能）

### 触覚的要素
- **glass-shadow: `0 8px 32px rgba(0, 0, 0, 0.06)`**: 軽い影、white bg の上で目立ちすぎない。KEEP
- **glass-shadow-modal: `0 8px 40px rgba(0, 0, 0, 0.12)`**: modal 専用の強めシャドウ。KEEP
- **glass-border-soft/faint**: `rgba(0, 0, 0, 0.08 / 0.15)` 黒ベースの枠。3 repo は accent 色枠、pd は中性黒枠。KEEP — 明確な差分
- **about-trigger の `i` ボタン**: awareness と共通のインタラクション pattern（awareness の about-overlay と同一系譜）。SYSTEMIZE（pd と awareness で共有可能、もしくは awareness が pd から継承した事実）

### インタラクション
- **4 modal を属性駆動で制御**（`data-modal-open="about/knowledge/howto/slides"` / `data-modal-close="..."`）: Bootstrap 非依存の軽量 modal。KEEP — **SYSTEMIZE 最優先（全 repo 展開候補）**
- **knowledge-modal は URL param `?knowledge=design-thinking` 駆動**: 調査ノート（デザイン思考 / 信頼 / 価値）を md lazy load。KEEP — **connector 再構成可能性（観点C）に直結**
- **howto-modal は「LLM に読ませる」ガイド**: pd が **自身を LLM 協働のスタート地点**として提示。3 ステップ + 調査 / ハーネス / 思考整理の 3 パターン + Tips + ライセンス。KEEP — pd 独自の価値提案
- **slides-modal は canvas renderer**: プレゼン資料を pd 内から直接表示。KEEP
- **lang-toggle（JA/EN）**: `data-lang-button` 属性で切替。KEEP
- **copy-target pattern**（`data-copy-target="prompt-basic"` + クリップボード copy）: プロンプト例をコピーさせる。SYSTEMIZE — 他 repo にも応用可
- **`?dev` param で glow panel 有効化**（推定、tokens.css のコメントから）: live token tuning。KEEP

### index.html 特有の所見
- **3 repo が dark theme + accent 青 + 明朝/mono なのに対し、pd は light theme + warm palette + glow**: pd と 3 repo は**別ブランドレイヤー**。pd を「理論の上位」と位置づけるメタファーとして、視覚的に反転している
- **howto-modal で pd 自身を LLM 用 input として宣伝**: ハーネス / ナレッジベースとしての自己言及。design system の「LLM 可食性」としても重要
- **knowledge-modal の URL param 駆動は docs/Slack connector の雛形**: 外部から `?knowledge=xxx` で呼べる → connector が URL を構成するだけで pd の md に辿り着ける

## pd 特有の観点（observation C の重要項目）

### quartz 層との関係

- **index.html と quartz/ は完全に別の技術スタック**:
  - index.html: Vanilla JS + 独自 CSS（tokens.css + styles/*）
  - quartz/: TypeScript + Preact/JSX + Quartz 標準 theme
- **視覚言語の乖離**:
  - index.html: light theme（`#fafaf9`）+ glow + warm palette（gold/navy/coral）
  - quartz/: Quartz 標準（darkmode toggle ありだが独自 palette）
- **共有コンポーネント**: `ProjectDesignTopNav` のみ（header 一本）
- **判断**: 意図的な **「広告層（index）と辞書層（quartz）」** の二重構造。視覚言語を unify するとブランド弱体化の恐れ。一方で design system 集約観点では**知見の断絶**が問題

### 上位参照性（design system の single source としての適性）

- **pd 独自の token 群**（`--bg / --ink / --gold / --navy / --coral / --glass-* / --glow-*`）と、3 repo 共通 token（`--color-accent / --kesson-* /`）が**非互換**
- **pd に design system を集約する場合の選択肢**:
  1. **pd の独自 token を捨てて 3 repo 側に寄せる**: pd の light theme を失う
  2. **pd は「メタレベル」の design system（`--ds-*`）を定義し、3 repo も pd 自身もそれを参照**: レイヤーが増える
  3. **pd に 3 repo の token を並行定義し、pd 自身は別 token を維持**: 重複増
- **推奨 B フェーズ判断**: (2) が最も柔軟。`--ds-core-*`（色/タイポ/radius/transition）+ `--ds-repo-scope-*`（repo 個性）の二段構造

### docs/Slack connector 再構成可能性（観点C 直結）

- **knowledge-modal の URL param 駆動**: `?knowledge=design-thinking` で md 表示 → connector が URL を組み立てるだけで pd の md に辿り着ける。**connector 雛形として理想的**
- **howto-modal の LLM ガイド**: pd 自身が「AI に読ませる」想定で構築済み。**docs connector との親和性が高い**
- **quartz の Graph 機構**: 概念間の双方向リンクを機械可読にする。**connector が concept graph を query できる**
- **Claude Design にこの構造を与える手順（推奨）**:
  - `?knowledge=xxx` URL を concept API として公開
  - quartz 層の graph を JSON-LD で export
  - howto-modal の prompt 例を Claude Design の seed prompt に流用

## サマリ

### 定量

- **KEEP 要素数**: 約 30
- **SYSTEMIZE 要素数**: 約 8
- **DROP 要素数**: 0（現状、惰性要素は見当たらない）
- **スコープ外**: quartz/public/*.html の個別 content（UI ではなく生成物）

### 全 repo 共通化できそうな要素（観点B で pd 集約優先）

1. **軽量 modal pattern（属性駆動: `data-modal-open / data-modal-close`）**: pd が 4 modal で実装済み、awareness の about-overlay も同系統。SYSTEMIZE 最優先で全 repo 展開
2. **URL param で md lazy load（`?knowledge=xxx`）**: connector 雛形、docs/Slack 連携の起点
3. **about-trigger（`i` button）**: pd と awareness で共有されている。SYSTEMIZE（全 repo で "about" の標準化）
4. **copy-target pattern**: クリップボードコピー UI、howto や code snippet 用、全 repo 応用可
5. **glow 表現**: pd 独自だが、全 repo で「runtime tunable token」として展開する価値あり

### 判断の非自明点

- **pd の visual language は 3 repo と非互換**: `#fafaf9` light theme vs `#050508` dark theme。**反転**は意図的で、放置すると集約の壁になる
- **pd に design system を集約 = pd の UI を変えないまま別レイヤーに置く必要**: `.claude/` や `docs/design-system/` など、pd 本体と別ディレクトリ / 別 brand で運用するのが現実的
- **i18n 方式の **三つ巴****:
  - kesson: `data-ja / data-en`（leaf 直書き）
  - creation / awareness: `data-i18n + data-i18n-attr-*`（key + dict、aria-label 含む）
  - pd: `data-i18n-key + data-i18n-aria-label-key`（key + 独立 aria key、creation/awareness と key 名規則が異なる）
  - 統一候補は creation/awareness 方式（aria 含めて dict で制御）。pd を寄せる場合 `data-i18n-aria-label-key` → `data-i18n-attr-aria-label` リネーム

- **glow を 3 repo に展開するか**: pd 独自ブランドを他 repo が inherit するのは「理論の上位」との整合になるが、全体のトーンを変える影響大

### 気づき（観点B/C で先に決めたほうが良いこと）

- **pd を design system 単一ソースとする場合、pd 本体の UI 刷新は**しない**方が良い**: pd の light theme + glow は brand identity、維持前提。単一ソース化のための **追加レイヤー**（`design-system/tokens.json` 相当）を別ディレクトリに構築する
- **quartz の Graph JSON export**: docs connector で pd の concept graph を機械可読にする手段、B/C フェーズの重要 feasibility 論点
- **howto-modal の prompt 例は design system の「LLM 向け仕様書」としても流用可**: Claude Design に食わせる seed として直接使える

### pd 集約時の論点（観点C で決定すべき）

1. **token prefix**: `--ds-*`（meta）/ `--pd-*`（pd 固有）/ `--kesson-* / --cs-* / --as-*`（repo 個性）の 3 層構造を正式化
2. **配置先**: `project-design/design-system/` ディレクトリ新設（既存 `quartz/` と `src/` を邪魔しない）
3. **参照方式**: 3 repo から submodule / git-subrepo / CDN / npm package のいずれか（B/C フェーズで決定）
4. **quartz との統合**: quartz の Obsidian Publish 向け theme を design system tokens で上書きするか、quartz theme を別扱いにするか
5. **Claude Design 連携口**: `design-system/tokens.json` + `design-system/components.md` で仕様を二重化し、Claude Design に URL で渡せる状態にする
